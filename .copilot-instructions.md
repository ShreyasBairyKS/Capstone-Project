## Project Overview

**VisionFood QAI** — Production line quality inspection for food/beverage using YOLOv11 + EfficientViT + multi-modal label verification (QR codes + OCR date checking).

**Architecture:** Operator sets product type (wrapper/can/box/transparent bottle) once at run start. Pipeline reads from `ProductionRun` on every frame. No per-frame classification overhead.

## Key Architecture Reminders

**Product Type:** Set once at run start (via RunSetup form). Operator chooses wrapper/can/box/transparent bottle. Pipeline reads from active ProductionRun document on every frame.

**Container Contents:** Separate from packaging (liquid/solid/powder). Affects damage severity — liquid=0.95 risk, solid=0.60 risk. Controls remediation decision weight.

**Conveyor Belt:** BBox IOU stability > 3 consecutive frames → emit for inspection. Debounce 15 frames. Fast belt fallback emits best-candidate + warning log.

**Latency Budget:** End-to-end 300ms. OCR is the only module allowed to exceed 100ms (it's opt-in per SKU profile).

## Why This Structure

**Product type = set once at run start:**
- Single conveyor line runs one SKU per shift (no mixed products)
- Per-frame classification adds model overhead + transition errors
- Operator knows before starting; setting type eliminates whole error class
- Pipeline reads from `ProductionRun` on every frame (no classifier)

**Container contents ≠ packaging shape:**
- Rigid can holds liquid (beer) OR solid (chickpeas)
- Damage severity differs based on contents: liquid=0.95 risk (spill), solid=0.60 risk (shelf life)
- Separate field makes severity logic clean and reusable

**Rigid box = any cuboidal cardboard:**
- Not just milk cartons — also cereal, biscuits, pasta, tea, medicine, confectionery
- Unambiguous name; UI labels provide operator clarity

**Conveyor frame selection (classical CV):**
- BBox IOU stability tracking (not frame differencing which picks up belt texture)
- Fast belt fallback prevents products exiting uninspected
- Debounce prevents re-inspection as product leaves frame

## Configuration Reference

| Setting | Type | Default | Purpose |
|---|---|---|---|
| `BELT_TRIGGER_MODE` | str | `"software"` | Frame selection: software IOU or (future) hardware sensor |
| `BELT_STABILITY_IOU_THRESHOLD` | float | `0.92` | Minimum bbox IOU between frames to count stable |
| `BELT_MIN_STABLE_FRAMES` | int | `3` | Consecutive stable frames before inspection trigger |
| `BELT_DEBOUNCE_FRAMES` | int | `15` | Frames to suppress after trigger (prevents re-trigger) |
| `BELT_TRIGGER_ROI` | list | `[0.25, 0.1, 0.75, 0.9]` | Normalised [x1, y1, x2, y2] of inspection region |
| `QR_VERIFICATION_ENABLED` | bool | `True` | Enable barcode verification |
| `QR_CACHE_TTL_SEC` | int | `300` | Per-SKU QR cache TTL |
| `OCR_ENABLED` | bool | `True` | Enable OCR date verification |
| `OCR_MIN_CONFIDENCE` | float | `0.6` | Minimum OCR extraction confidence |
| `DATE_MISMATCH_VERDICT` | str | `"ESCALATE"` | Verdict when printed date ≠ registered |
| `FILL_LEVEL_HARD_FAIL` | bool | `True` | Fill level deviation = FAIL (not confidence-gated) |

## Latency Budgets

| Component | Target | Rationale |
|---|---|---|
| End-to-end pipeline | < 300ms | Full inspection cycle |
| Frame selector | < 2ms | Called every frame at 30fps |
| Barcode verifier | < 10ms | pyzbar is C library (fast) |
| OCR verifier | < 800ms | CPU-bound Tesseract; opt-in per SKU |
| Food surface inspector | < 35ms | Classical CV only |
| Beverage inspector | < 30ms | Classical CV + Hough |

If a benchmark exceeds target, document the bottleneck and flag for trained model replacement in next sprint.
